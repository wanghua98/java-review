# 文件目录树结构方案设计说明书

**更新日期**：2026-07-17
**状态**：待评审

---

# 1 背景

当前系统使用邻接表模型存储目录结构（`file_directory` 表，`parent_id` 指向父节点），在应用层通过递归查询构建目录树和做循环检测。随着目录层级和数量增加，应用层递归的效率和代码复杂度成为瓶颈。本文档给出两种优化方案。

---

# 2 方案一：邻接表 + MySQL 8.0 递归 CTE

## 2.1 方案说明

保持现有 `file_directory` 表结构不变，将应用层的递归查询（`isDescendantOf()` 最大 100 层循环）替换为 MySQL 8.0 的 **递归公用表表达式（Recursive CTE）**，在数据库层面完成树形查询。

## 2.2 核心查询

### 查询目录树（从指定节点往下全量）

```sql
WITH RECURSIVE dir_tree AS (
    -- 锚点：从指定目录开始
    SELECT id, parent_id, name, sort, 0 AS depth
    FROM file_directory
    WHERE id = :rootId AND status = 1

    UNION ALL

    -- 递归：逐层向下查找子目录
    SELECT d.id, d.parent_id, d.name, d.sort, dt.depth + 1
    FROM file_directory d
    INNER JOIN dir_tree dt ON d.parent_id = dt.id
    WHERE d.status = 1
)
SELECT * FROM dir_tree ORDER BY depth, sort;
```

### 查询祖先链（从指定节点往上到根）

```sql
WITH RECURSIVE dir_ancestors AS (
    SELECT id, parent_id, name, 0 AS depth
    FROM file_directory
    WHERE id = :dirId

    UNION ALL

    SELECT d.id, d.parent_id, d.name, da.depth + 1
    FROM file_directory d
    INNER JOIN dir_ancestors da ON d.id = da.parent_id
    WHERE d.parent_id != 0
)
SELECT * FROM dir_ancestors ORDER BY depth DESC;
```

### 循环检测（判断 target 是否是 source 的后代）

```sql
WITH RECURSIVE dir_descendants AS (
    SELECT id FROM file_directory WHERE id = :sourceId AND status = 1
    UNION ALL
    SELECT d.id FROM file_directory d
    INNER JOIN dir_descendants dd ON d.parent_id = dd.id
    WHERE d.status = 1
)
SELECT EXISTS(
    SELECT 1 FROM dir_descendants WHERE id = :targetId
) AS is_descendant;
```

### 统计目录下所有文件（含子目录）

```sql
WITH RECURSIVE dir_tree AS (
    SELECT id FROM file_directory WHERE id = :dirId AND status = 1
    UNION ALL
    SELECT d.id FROM file_directory d
    INNER JOIN dir_tree dt ON d.parent_id = dt.id
    WHERE d.status = 1
)
SELECT COUNT(*) FROM file_info
WHERE parent_id IN (SELECT id FROM dir_tree) AND status = 1;
```

## 2.3 改动范围

| 改动项 | 说明 |
|--------|------|
| 表结构 | 不变 |
| FileDirectoryServiceImpl.isDescendantOf() | 替换为 CTE SQL，删除 100 层循环 |
| 新增 getAncestors() | 获取面包屑导航路径 |
| 新增 getSubTreeFilesCount() | 统计目录树下的所有文件 |
| 新增 getDirectoryTree() | 一次性加载完整目录树 |

## 2.4 适用场景

目录层级深（10+ 层）且目录数量较大，需要频繁做树形查询和循环检测。

# 3 方案二：闭包表（Closure Table）

## 3.1 方案说明

新增一张 `file_directory_closure` 表，记录目录树中**所有**祖先-后代关系（包含自身关联），将树形查询转为扁平表查询。

### 现有表（不变）

```sql
-- 目录节点表（仅存节点本身信息，不再存储树关系以外的逻辑）
file_directory (
    id BIGINT PK,
    name VARCHAR(255),
    sort INT,
    create_user BIGINT,
    status TINYINT,
    create_time DATETIME,
    update_time DATETIME
    -- 注意：移除了 parent_id 字段，或保留作为冗余
)
```

### 新增表

```sql
CREATE TABLE file_directory_closure (
    ancestor_id   BIGINT NOT NULL COMMENT '祖先节点ID',
    descendant_id BIGINT NOT NULL COMMENT '后代节点ID',
    depth         INT    NOT NULL COMMENT '层级深度（0=自身，1=父，2=祖父...）',
    PRIMARY KEY (ancestor_id, descendant_id),
    INDEX idx_descendant (descendant_id),
    FOREIGN KEY (ancestor_id) REFERENCES file_directory(id),
    FOREIGN KEY (descendant_id) REFERENCES file_directory(id)
);
```

### 数据示例

```
目录结构：
 项目资料 (id=1)
   ├── 合同 (id=2)
   │    └── 采购合同 (id=3)
   └── 图片 (id=4)

file_directory_closure 记录：
| ancestor_id | descendant_id | depth |
|-------------|---------------|-------|
| 1           | 1             | 0     |  -- 自身
| 1           | 2             | 1     |  -- 子目录
| 1           | 3             | 2     |  -- 孙目录
| 1           | 4             | 1     |  -- 子目录
| 2           | 2             | 0     |  -- 自身
| 2           | 3             | 1     |  -- 子目录
| 3           | 3             | 0     |  -- 自身
| 4           | 4             | 0     |  -- 自身
```

## 3.2 核心查询

### 查询所有子目录（含多级）

```sql
SELECT d.* FROM file_directory d
INNER JOIN file_directory_closure c ON d.id = c.descendant_id
WHERE c.ancestor_id = :dirId AND c.depth > 0 AND d.status = 1
ORDER BY c.depth, d.sort;
```

### 查询祖先链（面包屑）

```sql
SELECT d.* FROM file_directory d
INNER JOIN file_directory_closure c ON d.id = c.ancestor_id
WHERE c.descendant_id = :dirId
ORDER BY c.depth DESC;
```

### 循环检测（O(1)）

```sql
SELECT EXISTS(
    SELECT 1 FROM file_directory_closure
    WHERE ancestor_id = :sourceId AND descendant_id = :targetId AND depth > 0
) AS is_descendant;
```

### 统计目录下所有文件

```sql
SELECT COUNT(*) FROM file_info f
INNER JOIN file_directory_closure c ON f.parent_id = c.descendant_id
WHERE c.ancestor_id = :dirId AND f.status = 1;
```

## 3.3 DML 改动

### 创建目录

```sql
-- 1. 插入节点
INSERT INTO file_directory (name, sort, create_user, status) VALUES (...);

-- 2. 插入闭包关系（新节点 + 继承父节点的所有祖先）
INSERT INTO file_directory_closure (ancestor_id, descendant_id, depth)
SELECT ancestor_id, :newId, depth + 1
FROM file_directory_closure
WHERE descendant_id = :parentId
UNION ALL
SELECT :newId, :newId, 0;
```

### 删除目录（软删除）

```sql
-- 节点表和闭包表都只做逻辑删除，或物理删除闭包记录
DELETE FROM file_directory_closure
WHERE descendant_id IN (
    SELECT descendant_id FROM (
        SELECT descendant_id FROM file_directory_closure
        WHERE ancestor_id = :dirId
    ) AS sub
);
UPDATE file_directory SET status = 0 WHERE id = :dirId;
-- 递归删除文件的逻辑不变
```

### 移动目录

```sql
-- 1. 删除该节点及其所有后代与外部祖先的关系
--    （先断开节点子树与所有祖先的关联）
DELETE FROM file_directory_closure
WHERE descendant_id IN (
    SELECT descendant_id FROM (
        SELECT descendant_id FROM file_directory_closure
        WHERE ancestor_id = :dirId
    ) AS sub
)
AND ancestor_id IN (
    SELECT ancestor_id FROM (
        SELECT ancestor_id FROM file_directory_closure
        WHERE descendant_id = :dirId AND depth > 0
    ) AS sub
);

-- 2. 重新建立与新父目录的关联
--    （节点子树关联到新父目录的所有祖先，加上新父目录自身）
INSERT INTO file_directory_closure (ancestor_id, descendant_id, depth)
SELECT super.ancestor_id, sub.descendant_id, super.depth + sub.depth + 1
FROM file_directory_closure AS super
CROSS JOIN file_directory_closure AS sub
WHERE super.descendant_id = :newParentId
  AND sub.ancestor_id = :dirId;
```

## 3.4 改动范围

| 改动项 | 说明 |
|--------|------|
| 新增表 | file_directory_closure |
| file_directory 表 | 可保留 parent_id 做冗余，也可移除 |
| FileDirectoryServiceImpl | 所有查询改为查闭包表，新增/删除/移动需同步维护闭包 |
| 数据迁移 | 现有目录数据需要刷入闭包表（一次性脚本） |

## 3.5 适用场景

目录树**查询频繁**（特别是循环检测和统计），而写入操作（创建/移动目录）相对较少。

# 4 方案对比

| 对比维度 | 邻接表 + CTE | 闭包表 |
|---------|-------------|--------|
| 新增表 | 无 | 1 张额外表 |
| 表数据量 | N 行 | N² 行（最坏情况） |
| 查询子目录树 | ⚡ 递归 CTE，数据库层遍历 | ⚡ 单表等值 JOIN |
| 循环检测 | ⚡ 递归 CTE | ⚡ O(1) 查闭包表 |
| 祖先链（面包屑） | ⚡ 递归 CTE | ⚡ 单表等值 JOIN |
| 统计文件数 | ⚡ 递归 CTE + COUNT | ⚡ 单表 JOIN + COUNT |
| 创建目录 | ⚡ 单条 INSERT | ⚡ 批量 INSERT（继承父节点祖先） |
| 删除目录 | ⚡ 递归软删除（不变） | ⚠️ 需要批量 DELETE 闭包关系 |
| 移动目录 | ⚠️ 递归 CTE 循环检测 | ⚠️ 需要 DELETE + INSERT 重建闭包关系 |
| 代码复杂度 | 低，移除应用层递归即可 | 中，DML 逻辑变复杂 |
| 数据库版本要求 | MySQL 8.0+ | MySQL 5.7+ 即可 |

## 4.1 推荐建议

- **短期（当前版本）**：继续使用邻接表，将应用层递归改为 **MySQL 8.0 CTE**（方案一）。改动最小，仅涉及查询语句替换，不影响表结构和 DML。
- **长期（目录操作频繁且规模增大）**：迁移到 **闭包表**（方案二）。查询性能最优，但建表和 DML 复杂度增加。
