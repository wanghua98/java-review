## 6\.8 过期任务清理机制

上传任务如果中途放弃（status=0 且长时间未更新），临时分片文件会占用磁盘空间。

清理策略：

- 定时任务每日凌晨扫描超过 **24小时** 未完成的上传任务

- 删除该任务对应的临时分片文件和 `file_chunk` 记录

- 将任务状态更新为 2（上传失败）

- 清理 Redis 中对应的上传进度缓存

### 秒传边界说明

秒传功能存在以下边界场景需要处理：

- **不同用户上传相同文件**：秒传时为当前用户新建一条 `file_info` 记录，引用同一物理文件（`storage_path` 相同），避免文件重复存储

- **文件归属目录不同**：即使物理文件相同，不同用户的文件记录 `parent_id` 可以不同，互不影响

- **并发秒传**：两个用户同时上传相同文件时，通过数据库唯一索引（`uk_file_sha256`）避免重复创建任务

# 8 目录管理设计

## 8\.1 设计目标

为了方便用户对附件进行分类管理，系统提供目录管理功能。

目录支持：

- 新建目录 

- 编辑目录 

- 删除目录 

- 移动目录 

- 文件归档 

- 无限层级目录 

系统采用树形目录结构，实现类似 Windows 文件夹或网盘目录管理方式。

## 8\.2 目录结构设计

例如：

```Plain Text
全部文件

├── 项目资料
│
├── 合同
│
│    ├── 采购合同
│    │
│    ├── 销售合同
│
├── 图片
│
│    ├── 产品图片
│    ├── 宣传图片
│
└── 视频
```

用户可以自由创建目录。

目录下可以继续创建目录。

没有层级限制。

## 8\.3 目录树设计

系统采用父子节点方式保存目录。

例如：

```Plain Text
根目录(id=1)

│

├── 项目(id=2)

│      │

│      ├── 合同(id=3)

│      │

│      └── 图片(id=4)

│

└── 视频(id=5)
```

每个目录保存自己的父节点编号。

通过递归即可构建完整目录树。

## 8\.4 新建目录

流程：

```Plain Text

↓

选择父目录

↓

↓

```

如果父目录为空，则表示创建一级目录。

## 8\.5 修改目录

允许修改：

```Plain Text

目录排序

```

目录编号保持不变。

## 8\.6 删除目录

系统采用逻辑删除。

删除前需要判断：

```Plain Text
目录下是否存在子目录

目录下是否存在文件
```

若存在，则禁止删除。

例如：

移动后目录树自动刷新。

每个文件只属于一个目录。

文件上传时可指定目录。

例如：

```Plain Text

↓

↓

```

查询目录时即可显示对应文件。

## 8\.9 无限级目录实现

系统采用邻接表（Adjacency List）模型实现无限级目录。

目录表仅保存当前节点及父节点信息。

例如：

查询目录树时，通过递归构建完整树结构。

其特点如下：

- 支持无限层级目录 

- 数据结构简单 

- 新增、移动目录成本低 

- 易于维护 

- 适用于附件管理、网盘等业务场景

## 8\.10 目录权限设计

目录按用户隔离，每个用户只能看到和操作自己创建的目录和文件。

- **查询**：查询目录树时按 `create_user` 过滤，用户只能看到自己的目录

- **创建**：用户只能在自己拥有的目录下创建子目录

- **删除/移动**：只能操作自己创建的目录

- **文件归属**：文件跟随上传用户，不同用户即使上传相同文件也各自独立

管理员（ADMIN角色）可查看所有用户的文件和目录，用于系统管理。

# 9 数据库设计

## 9\.1 数据库设计概述

附件文件管理系统作为一个独立的文件服务平台，需要同时满足**用户文件管理**和**业务系统附件调用**两类场景。

系统数据库主要用于保存：

- 用户基础信息 

- 文件上传任务信息 

- 文件分片上传状态 

- 文件基本信息 

- 目录信息 

- 文件操作行为记录 

由于系统支持大文件分片上传，因此数据库设计需要区分：

1. **上传过程数据**

2. 用于记录文件上传过程中的任务状态、分片情况。

3. **文件正式数据**

4. 用于保存上传完成后的文件信息，并提供后续查询、下载、预览服务。

系统数据库整体关系如下：

```Plain Text
用户信息

   | 1:N

   |
目录信息 ←── 自身关联（parent_id）
   |
   | 1:N
   |
上传任务
   |
   | 1:N
   |
文件分片


上传完成


   |
   | (parent_id → 目录)
   |
文件信息
   |
   | 1:N
   |
操作日志
```

系统数据库主要包含以下数据表：

- sys_user：用户信息表
- file_directory：目录表
- file_upload_task：文件上传任务表
- file_chunk：文件分片信息表
- file_info：文件信息表
- file_operation_log：文件操作日志表

- 1：正常

- 0：已删除

删除前需校验目录下是否存在子目录或文件，若存在则禁止删除。

## 9\.8 数据库关系设计

系统主要数据关系如下：

```Plain Text
sys_user
    |
    | 1:N
    |
file_directory
    |
    | 1:N (自身关联，parent_id)
    |
    └── 子目录


sys_user
    |
    | 1:N
    |
file_upload_task
    |
    | 1:N
    |
file_chunk


file_upload_task (parent_id)
    →  file_directory (id)


上传完成


sys_user
    |
    | 1:N
    |
file_info
    |
    | N:1
    |
file_directory (parent_id)


file_info
    |
    | 1:N
    |
file_operation_log
```

关键关系说明：

1. sys_user → file_directory：用户创建目录，1:N关系

2. file_directory → file_directory：通过parent_id实现自身关联，支持无限层级目录

3. sys_user → file_upload_task：用户创建上传任务，1:N关系

4. file_upload_task → file_chunk：一个任务对应多个分片，1:N关系

5. file_upload_task → file_directory：上传任务通过parent_id关联目标目录

6. sys_user → file_info：用户上传文件，1:N关系

7. file_info → file_directory：文件通过parent_id关联所属目录

8. file_info → file_operation_log：一个文件可有多条操作日志，1:N关系
