### 3\.2\.1 为什么选择这些技术

- **分片状态暂存 Redis 而非直接写数据库**：大文件分片上传过程中，每个分片都直接写入 MySQL 会产生大量写操作。实际设计中，分片信息先暂存到 Redis Hash，仅在合并触发时一次性批量写入数据库，大幅减少 DB 压力。

- **Sa\-Token 而非 Spring Security**：Sa\-Token 更轻量，API 简洁，天然支持 Token 认证，适合内部系统。

- **Lettuce 而非 Jedis**：Lettuce 支持异步、连接池更成熟、线程安全。

# 6 分片上传模块设计（核心）

## 6\.1 设计目标

针对大文件上传过程中的：

- 网络不稳定导致上传中断

- 上传时间过长

- 失败后需要重新开始

等问题，实现可靠的大文件上传。

支持：

- 文件切片（客户端按 5MB 切割）

- 分片并发上传（浏览器并发控制）

- 断点续传（Redis 记录已上传分片编号）

- 秒传（SHA\-256 查重）

- 服务端自动合并

- 异步哈希完整性校验

## 6\.2 整体上传流程

```Plaintext
用户选择文件
     │
     ▼
前端计算 SHA-256（使用 hash-wasm 分块计算）
     │
     ▼
调用 POST /api/file/init 初始化上传任务
     │
     ├── SHA-256 已存在 ──► 秒传：直接返回文件信息
     │
     ├── 存在未完成任务（同一用户+同一SHA-256+status=0）
     │    └── Redis 中有记录 ──► 续传：返回已上传分片编号列表
     │
     └── 新文件 ──► 创建上传任务（status=0），分片总数写入 Redis
     │
     ▼
前端根据 taskId 和 chunkSize（5MB）自动文件切片
     │
     ▼
循环调用 POST /api/file/upload 上传各分片
（参数：taskId + chunkNumber + file 二进制）
     │
     ├── Redis 判断分片是否已上传（跳过已上传分片）
     │
     └── 保存分片到磁盘 → 写入 Redis Hash → 异步更新 DB 计数
     │
     ▼
最后一个分片上传完成后（Redis HLEN == chunkCount）
     │
     ├── 获取 Redis 分布式锁（防止并发合并）
     ├── 从 Redis 批量读取分片信息 → 批量写入 DB
     ├── 清理 Redis 分片缓存
     ├── 使用 FileChannel.transferTo 合并 .part 文件
     ├── 插入 file_info 记录（status=1 上传成功未计算哈希）
     ├── 更新任务 status=2（已合并）
     └── 事务提交后，异步计算 SHA-256 并与前端对比
         ├── 匹配 → status=3 哈希正常, 写入 file_sha256
         └── 不匹配 → status=-1, 文件名追加"（文件损坏）"
```

## 6\.4 上传初始化（秒传/续传判断）

**接口**：`POST /api/file/init`

**前端提交**：

```JSON
{
  "fileSha256": "xxxx...",      // 可选，前端用 hash-wasm 计算
  "fileName": "test.zip",
  "fileSize": 104857600,
  "fileSuffix": "zip",
  "chunkCount": 20,
  "parentId": 1                // 可选，不传默认为0（根目录）
}
```

**后端处理流程**（`FileUploadTaskServiceImpl.initFile()`）：

```Plaintext
接收请求
  │
  ├─ SHA-256 非空？
  │    │
  │    ├─ 查询 file_info 是否存在相同 SHA-256 ──► 秒传：
  │    │     创建新 file_info 记录（引用同一 storage_path）
  │    │     返回 taskId = -1（前端无需上传）
  │    │
  │    └─ 查询是否有未完成的上传任务
  │       （同一用户 + 同一 SHA-256 + status=0）
  │          │
  │          ├─ 存在 且 Redis 中有记录（isTaskExists）──► 续传：
  │          │    从 Redis 读取已上传分片编号集合
  │          │    返回给前端（前端跳过已上传的分片）
  │          │
  │          └─ 不存在 ──► 创建新任务（status=0）
  │
  └─ SHA-256 为空 ──► 创建新任务
```

**返回示例**：

```JSON
// 新任务
{
  "code": 200,
  "message": "success",
  "data": {
    "id": 42,           // 任务ID
    "chunkCount": 20,
    "uploadedCount": 0,
    "status": 0
  }
}

// 续传
{
  "code": 200,
  "message": "success",
  "data": {
    "id": 42,
    "uploadedChunks": [1, 2, 5, 6],  // 已上传分片
    "chunkCount": 20,
    "uploadedCount": 4,
    "status": 0
  }
}

// 秒传
{
  "code": 200,
  "message": "success",
  "data": {
    "id": -1            // -1 表示秒传成功，无需上传
  }
}
```

## 6\.5 分片上传（核心）

**接口**：`POST /api/file/upload`

**参数**：

|参数|类型|必填|说明|
|---|---|---|---|
|taskId|Long|是|上传任务ID（从 init 返回）|
|chunkNumber|Integer|是|分片编号（从1开始）|
|file|MultipartFile|是|分片文件二进制数据|

**分片接收流程**（`FileChunkServiceImpl.saveChunk()`）：

```Plaintext
接收分片请求
  │
  ├─ 1. Redis 校验任务是否存在（isTaskExists）
  │     └─ 不存在 → 返回错误"上传任务不存在"
  │
  ├─ 2. 创建分片目录 {uploadPath}/{taskId}/
  │
  ├─ 3. Redis 判重（isChunkUploaded）
  │     └─ 已上传 → 直接返回成功（幂等，前端可能重传）
  │
  ├─ 4. 分片写入磁盘 {chunkNumber}.part
  │     └─ file.transferTo() → {uploadPath}/{taskId}/{chunkNumber}.part
  │
  ├─ 5. 分片信息暂存 Redis Hash
  │     key: "chunk:task:{taskId}"
  │     field: chunkNumber.toString()
  │     value: FileChunk 的 JSON（含 taskId, chunkNumber, chunkPath）
  │     expire: 1 天
  │
  ├─ 6. 异步更新 DB uploaded_count（setIncrBy +1）
  │     └─ 仅做记录用途，合并条件不由 DB 判断
  │
  ├─ 7. 通过 Redis 判断是否全部分片就绪
  │     uploadedCount = HLEN("chunk:task:{taskId}")
  │     totalCount = GET("chunk:task:{taskId}:cnt")
  │     │
  │     ├─ uploadedCount < totalCount → 返回"分片保存成功"
  │     │
  │     └─ uploadedCount >= totalCount → 执行合并
  │
  └─ [合并流程]（详见 6.6）
```

### Redis 数据结构设计

```Plaintext
# 任务分片总数（init 时写入）
chunk:task:{taskId}:cnt  →  "20"    (String, TTL=1天)

# 分片信息 Hash（每上传一个分片写入一个 field）
chunk:task:{taskId}  →  Hash (TTL=1天)
  field: "1"  →  '{"taskId":42,"chunkNumber":1,"chunkPath":"1.part"}'
  field: "2"  →  '{"taskId":42,"chunkNumber":2,"chunkPath":"2.part"}'
  ...

# 分布式锁（合并时使用）
lock:merge:{taskId}  →  "clientId"  (String, TTL=10分钟)
```

**为什么分片信息暂存 Redis 而不是直接写数据库？**

1. **减少 DB 写入**：1000 个分片的任务，不需要写 1000 次 file\_chunk 表，只在合并时一次性批量写入

2. **O\(1\) 判重**：通过 Redis Hash field exists 判断分片是否已上传，无需查 DB

3. **快速判断完成**：`HLEN` 对比 `chunkCount` 是 O\(1\) 操作，比 DB COUNT 快一个数量级

4. **续传查询快**：直接从 Hash 的 field keys 获取已上传分片编号

## 6\.6 文件合并（自动触发）

当 `HLEN("chunk:task:{taskId}") >= GET("chunk:task:{taskId}:cnt")` 时自动触发合并。

### 合并流程

```Plaintext
合并触发
  │
  ├─ 获取 Redis 分布式锁 "merge:{taskId}"
  │   ├─ 获取失败 → 返回"分片保存成功"（另一个线程正在合并）
  │   └─ 获取成功 → 继续
  │
  ├─ 从 Redis Hash 读取全部分片信息
  │   └─ 批量写入 file_chunk 数据库（saveBatch）
  │
  ├─ 清理 Redis 分片缓存（deleteChunkInfos）
  │
  ├─ 合并分片文件
  │   │
  │   ├─ 扫描目录下所有 .part 文件
  │   ├─ 按 chunkNumber 升序排序
  │   ├─ FileChannel.transferTo() 循环追加写入
  │   │    注意：transferTo 可能不会一次传完，
  │   │    需要 while 循环直到全部传输完成
  │   └─ 合并后的文件以 5 位随机数命名（如 "72119"）
  │
  ├─ 插入 file_info 记录（status=1）
  │   ├─ fileName: 调用 resolveUniqueFileName() 自动重命名去重
  │   ├─ storagePath: "{taskId}/{random5digits}"
  │   ├─ fileSize: 实际文件大小
  │   ├─ fileSha256: null（尚未计算）
  │   └─ status: 1（上传成功未计算哈希）
  │
  ├─ 更新任务 status=2（已合并待计算哈希）
  │
  └─ 注册事务提交后回调 AfterCommitRunner
      └─ 异步线程池执行 computeAndVerifyHash()
```

### 合并代码关键实现

```Java
// FileChannel.transferTo 循环拼接
try (FileOutputStream fos = new FileOutputStream(finalFilePath);
     FileChannel outChannel = fos.getChannel()) {
    for (File chunk : chunks) {
        try (FileInputStream fis = new FileInputStream(chunk);
             FileChannel inChannel = fis.getChannel()) {
            long size = inChannel.size();
            long transferred = 0;
            // 文件可能一次不能一次传完，循环等待传输完成
            while (transferred < size) {
                transferred += inChannel.transferTo(
                    transferred, size - transferred, outChannel);
            }
        }
    }
}
```

### 并发安全设计

|场景|解决方案|
|---|---|
|两个分片同时触发合并|Redis 分布式锁，只让一个线程执行合并|
|锁意外未释放|TTL 10 分钟自动释放|
|误删别人的锁|Lua 脚本验证 clientId 后才删除|
|合并失败文件不完整|catch 块删除不完整文件并 throw 异常回滚事务|

## 6\.7 异步哈希校验

合并完成后，通过 `AfterCommitRunner.afterCommit()` 注册事务提交后的回调，在异步线程池中计算 SHA\-256。

```Java
AfterCommitRunner.afterCommit(() -> executorService.execute(() ->
    computeAndVerifyHash(taskId, finalStoragePath, fullPath,
                         uploadTask.getFileSha256())));
```

### 计算方式

使用 `MappedByteBuffer` 分块计算，每块 100MB，避免大文件 OOM：

```Java
try (RandomAccessFile raf = new RandomAccessFile(filePath, "r");
     FileChannel channel = raf.getChannel()) {
    long position = 0;
    long mapSize = Math.min(size, 100 * 1024 * 1024L);
    
    while (position < size) {
        long actualMapSize = Math.min(mapSize, size - position);
        MappedByteBuffer buffer = channel.map(
            FileChannel.MapMode.READ_ONLY, position, actualMapSize);
        //更新hash值
        digest.update(buffer);
        //更新指针
        position += actualMapSize;
    }
}
```

### 校验结果处理

|结果|file\_info\.status|文件名|说明|
|---|---|---|---|
|哈希匹配|3|不变|正常可用，写入 SHA\-256|
|哈希不匹配|\-1|追加"（文件损坏）"|传输过程中数据可能损坏|
|前端未传 SHA\-256|1|不变|不做校验，保持未计算状态|

## 6\.8 秒传设计

### 流程

```Plaintext
前端计算 SHA-256
     │
     ▼
POST /api/file/init (携带 fileSha256)
     │
     ▼
后端查询 file_info 是否存在相同 file_sha256
     │
     ├─ 存在 ──► 创建新 file_info 记录
     │   │         │
     │   │         ├─ fileName: 使用前端上传的文件名（自动重名名去重）
     │   │         ├─ storagePath: 引用已有的物理文件路径
     │   │         ├─ fileSize: 从已有记录复制
     │   │         ├─ createUser: 当前用户
     │   │         └─ parentId: 用户指定的目录
     │   │
     │   └─ 返回 taskId = -1（前端不触发实际上传）
     │
     └─ 不存在 ──► 正常创建上传任务，走分片上传流程
```

### 边界场景

|场景|处理方式|
|---|---|
|不同用户上传相同文件|每个用户各自创建一条 file\_info，引用同一 storage\_path|
|秒传到不同目录|parent\_id 不同，互不影响|
|并发秒传|通过数据库索引保证不会重复创建物理文件|
|同名文件冲突|自动追加 \(1\)、\(2\) 等编号|

## 6\.9 过期任务清理

> **v3 说明**：当前版本尚未实现过期任务定时清理功能。

**设计预期**：

- 上传任务若中途放弃（status=0 且长时间未更新），临时分片文件会占用磁盘空间

- 清理策略：定时任务（如 Cron）每日凌晨扫描超过 24 小时未完成的 upload\_task → 删除临时分片文件和 file\_chunk 记录 → 任务状态更新为 2（失败）→ 清理 Redis 缓存

# 7 目录管理模块设计

## 7\.1 设计目标

方便用户对附件进行分类管理，系统提供完整的目录管理功能。

支持：

- 新建目录（自动解决同名冲突）

- 修改目录（重命名，带格式校验）

- 删除目录（递归软删除子目录和文件）

- 移动目录（含循环检测，最多追溯 100 层）

- 无限层级目录（邻接表模型）

## 7\.2 目录树结构（实际实现）

```Plaintext
根目录 (id=1, parent_id=0, name="admin")
  │
  └── User 父目录 (id=2, parent_id=1, name="User")
       │
       ├── 用户A的个人目录 (id=3, parent_id=2, name=用户名)
       │   ├── 用户A创建的子目录1
       │   └── 用户A创建的子目录2
       │
       └── 用户B的个人目录 (id=4, parent_id=2, name=用户名)
           └── 用户B创建的子目录
```

**目录结构说明**：

- 系统启动时由 `InitRunner` 自动创建根目录和 User 父目录

- 用户注册时自动在 User 父目录下创建以用户名为名称的个人目录

- 用户的所有操作都在其个人目录下进行

- `parent_id=0` 表示根目录（不可删除）

- `id=2` 的 User 父目录受保护（不可移动、不可重命名、不可删除）

- 用户个人目录受保护（不可删除）

## 7\.3 目录与文件合并分页

系统采用 **目录 \+ 文件合并计数参与分页** 的设计。

```Java
// FileDirectoryServiceImpl.getDirectoryContents()
// 1. 查询所有子目录（全部返回，按 sort 正序）
List<FileDirectory> allSubDirs = list(..."parent_id"=dirId, "status"=1, orderBy sort)

// 2. 查询文件总数
Long fileTotal = selectCount(..."parent_id"=dirId, "status"=1)

// 3. 总条目 = 子目录数 + 文件数，计算总页数
totalCount = allSubDirs.size() + fileTotal

// 4. 计算当前页目录偏移
dirStart = min(offset, allSubDirs.size())
dirEnd = min(offset + size, allSubDirs.size())

// 5. 计算文件查询偏移量（目录占位后剩下的行数分给文件）
fileQueryOffset = max(0, offset - allSubDirs.size())
fileQuerySize = size - dirCountOnPage

// 6. 查询文件（手动 LIMIT offset, size）
selectList(..."parent_id"=dirId, "status"=1, orderBy create_time DESC, LIMIT offset, size)
```

**响应结构**（`DirectoryVO`）：

```JSON
{
  "currentDir": { "id": 3, "parentId": 2, "name": "个人目录", ... },
  "subDirs": [ /* 子目录列表，按 sort 排序 */ ],
  "files": [ /* 文件列表，按创建时间倒序，当前页 */ ],
  "currentPage": 1,
  "pageSize": 20,
  "totalCount": 25,
  "totalPages": 2
}
```

## 7\.4 目录操作安全设计

|操作|保护规则|
|---|---|
|创建|校验父目录存在，同级自动重名名去重|
|删除|不允许删除根目录、User父目录、用户个人目录；递归软删除|
|移动|循环检测 `isDescendantOf()`，最多追溯100层|
|重命名|不允许重命名 id=2 的 User 父目录|
|所有权|所有操作校验 `createUser == userId`|

# 8 用户认证模块设计

## 8\.1 认证方式

采用 Sa\-Token Token 认证机制。

- **Token 存储**：Cookie（前端自动携带 credentials: 'include'）

- **Token 风格**：simple\-uuid

- **Token 有效期**：3600 秒（1 小时）

- **多点登录**：允许（is\-concurrent: true）

- **Token 共享**：不共享（每次登录新建一个 token）

## 8\.2 密码加密

采用 BCrypt \+ Pepper 加密方案。

```Plaintext
encode(password) = bcrypt("uniplore" + password)
matches(password, encoded) = bcryptCheck("uniplore" + password, encoded)
```

- **Pepper**："uniplore" 前缀，代码硬编码（非数据库存储）

- **BCrypt**：Hutool 的 `DigestUtil.bcrypt()` 实现

## 8\.3 注册流程

```Plaintext
接收注册请求
  │
  ├─ ValidateUtil 校验用户名格式（字母开头，字母数字组合，1~50字）
  ├─ ValidateUtil 校验密码长度（≥2位）
  ├─ 查询用户名是否已存在
  ├─ 创建用户
  │   ├─ password = PasswordUtil.encode(password)
  │   ├─ role = "USER"
  │   └─ nickname = "user_" + 5位随机数
  │
  ├─ 创建用户个人目录（createUserDirectory）
  │
  └─ 自动登录（StpUtil.login）
```

## 8\.4 角色区分

|角色|常量|说明|
|---|---|---|
|ADMIN|"ADMIN"|管理员（系统启动时默认创建 admin/aa）|
|USER|"USER"|普通用户（注册时自动分配）|

> **当前实现中，角色仅用于 Sa\-Token 权限接口返回，尚未实现基于角色的细粒度权限控制。**

# 9 操作日志模块设计

## 9\.1 设计方式

采用 **Spring AOP \+ 自定义注解** 的方式自动记录操作日志。

**注解**：

```Java
@Target({ElementType.METHOD, ElementType.TYPE})
@Retention(RetentionPolicy.RUNTIME)
public @interface LogRecord {
    String value() default "";
}
```

**切面**：

```Java
@Around("@annotation(logRecord)")
public Object around(ProceedingJoinPoint joinPoint, LogRecord logRecord) {
    Object result = joinPoint.proceed();  // 先执行方法
    // 记录日志
    FileOperationLog log = new FileOperationLog();
    log.setUserId(getCurrentUserId());
    log.setOperationType("API");
    log.setDescription(logRecord.value());
    fileOperationLogMapper.insert(log);
    return result;
}
```

## 9\.2 日志记录时机

- **先执行方法，后记录日志**：确保方法执行成功后才记录，避免失败操作也被记录

- **未登录时 userId 记录为 0**：通过 try\-catch 处理

## 14\.3 大文件上传组件特性

- 浏览器端 hash\-wasm 分块计算 SHA\-256

- 5MB 分片大小

- 并发上传控制（浏览器原生的并发限制）

- 上传进度显示

- 暂停/继续上传

- 失败自动重试

- 秒传跳过（根据 init 返回 taskId=\-1）

- 续传跳过已上传分片（根据 init 返回 uploadedChunks）

## 14\.4 文件预览

集成 kkFileView 服务：

```Plaintext
http://192.168.2.42:8012/onlinePreview?url=...
```

API 接口也提供直接内联预览：

```Plaintext
GET /api/file/inline/{fileId}.{suffix}
```

通过 `Content-Disposition: inline` 让浏览器直接打开可预览文件类型。
