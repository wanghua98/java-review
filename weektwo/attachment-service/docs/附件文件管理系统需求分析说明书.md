# 1 引言

## 1.1 编写目的

本文档用于明确附件文件管理系统的建设目标、业务需求、功能需求以及非功能需求，为系统设计、开发、测试和验收提供依据。

附件文件管理系统旨在为用户提供统一的文件存储与管理能力，实现附件资源集中化管理，提高文件使用效率和安全性。

## 1.2 系统背景

随着企业信息化系统不断增加，各业务系统通常需要处理大量附件资源，例如：
- 项目资料
- 合同文件
- 图片资源
- 办公文档
- 用户资料
- 业务凭证

传统方式中，文件管理方式分散，容易造成：
- 文件上传不稳定（大文件经常中断）
- 文件重复保存造成空间浪费
- 管理困难，查找效率低
- 权限控制不统一

因此，需要建设统一的附件文件管理系统，提供可靠的大文件上传、文件集中存储管理和便捷的查找能力。

# 2 系统建设目标

## 2.1 统一文件管理

建立统一附件管理平台，对系统中的文件资源进行集中管理。

## 2.2 支持大文件可靠上传

针对大文件上传场景，解决网络不稳定导致上传中断、上传时间过长、失败后重新开始等问题，提供：

- 分片上传
- 断点续传
- 秒传（基于 SHA-256 文件去重）
- 文件完整性校验

## 2.3 提供文件服务能力

为用户提供：

- 文件上传
- 文件下载
- 文件预览
- 文件删除
- 文件重命名
- 文件移动
- 文件查询（按目录浏览）

## 2.4 提供目录管理能力

用户可自行管理文件目录结构，支持：

- 创建目录
- 修改目录
- 删除目录
- 移动目录
- 无限层级目录

## 2.5 文件安全性

- 用户隔离：用户只能看到和操作自己的文件和目录
- 用户认证：基于 Token 的登录认证
- 所有权校验：删除/移动/重命名等操作校验用户所有权
- 文件完整性：上传完成后异步计算 SHA-256 并与前端比对

# 3 系统用户角色分析

## 3.1 普通用户

普通用户是系统的主要使用者，主要用于个人文件管理。

权限：
- 注册/登录/登出系统
- 上传文件（支持大文件分片上传）
- 查看个人文件列表（按目录浏览，分页）
- 下载文件
- 预览文件（在线查看图片/文档等）
- 删除个人文件（软删除）
- 重命名个人文件
- 移动个人文件到其他目录
- 管理个人目录（创建/修改/删除/移动/重命名）

## 3.2 系统管理员

管理员负责系统初始化以及整体管理。

权限：
- 自动获得系统启动时创建
- 拥有普通用户的所有功能
- **注**：当前版本管理员不拥有特殊管理后台，管理能力有待后续扩展

# 4 系统总体业务流程

## 4.1 用户注册流程

```
用户填写用户名和密码
        ↓
系统校验用户名格式（必须以字母开头，仅包含字母数字）
        ↓
系统校验密码长度（至少 2 位）
        ↓
检查用户名是否已存在
        ↓
创建用户（BCrypt + Pepper 加密密码）
        ↓
为用户创建个人目录（以用户名为目录名）
        ↓
自动登录，跳转到文件管理页面
```

## 4.2 用户登录流程

```
用户输入用户名和密码
        ↓
系统查询用户是否存在
        ↓
校验密码（BCrypt + Pepper）
        ↓
生成 Token（Sa-Token）
        ↓
写入 Cookie
        ↓
跳转到文件管理页面
```

## 4.3 文件上传（大文件分片上传）流程

```
用户选择文件
        ↓
前端使用 hash-wasm 计算文件 SHA-256
        ↓
调用上传初始化接口（POST /api/file/init）
        │
        ├── SHA-256 已存在 ──► 秒传：直接完成，无需实际上传
        │
        ├── 存在未完成的上传任务 ──► 续传：返回已上传分片编号列表
        │                             前端跳过已上传的分片
        │
        └── 新文件 ──► 创建上传任务，返回 taskId 和总分片数
        │
        ▼
前端按 5MB 切割文件为多个分片
        │
        ▼
循环上传各分片（POST /api/file/upload）
  参数：taskId + chunkNumber + file
        │
        ├── Redis 判重：已上传的分片直接跳过
        └── 服务端保存分片文件到磁盘，信息暂存 Redis
        │
        ▼
服务端判断全部分片是否就绪（Redis HLEN == chunkCount）
        │
        ├── 否：返回"分片保存成功"，等待后续分片
        └── 是：自动触发合并流程
                │
                ├── Redis 分布式锁（防并发合并）
                ├── 批量写入分片信息到数据库
                ├── FileChannel.transferTo 拼接 .part 文件
                ├── 生成正式文件记录（file_info）
                └── 异步计算 SHA-256 并与前端传入值比对
```

## 4.4 目录文件浏览流程

```
用户登录系统
        ↓
进入文件管理页面
        ↓
系统查询用户个人目录
        ↓
加载目录下子目录和文件列表（合并分页）
        │
        ├── 子目录全部返回（按 sort 排序）
        └── 文件按创建时间倒序分页返回
        │
        ▼
用户可点击子目录进入，或通过面包屑导航返回上级目录
        │
        ▼
用户可对文件/目录执行下载、预览、删除、重命名、移动等操作
```

# 5 功能需求分析

## 5.1 用户认证管理

### 功能描述

系统需要提供用户身份认证功能，包括注册、登录、登出和信息管理。

### 功能要求

1. **用户注册**
   - 输入用户名和密码完成注册
   - 用户名格式：以字母开头，仅包含字母数字，长度 1~50
   - 密码长度：2~100 位
   - 注册后自动登录
   - 注册时自动为用户创建个人目录
   - 自动生成昵称（格式：user_ + 5位随机数）

2. **用户登录**
   - 输入用户名和密码登录
   - 密码使用 BCrypt + Pepper 加密验证
   - 登录成功后由 Sa-Token 生成 Token，写入 Cookie
   - Token 有效期 1 小时

3. **用户登出**
   - 销毁当前登录 Token
   - 清除登录状态

4. **用户信息查看**
   - 获取当前登录用户信息（密码脱敏显示）

5. **修改用户信息**
   - 支持修改用户名、密码
   - 修改用户名时同步更新个人目录名称
   - 修改用户名需校验唯一性

## 5.2 文件上传管理

### 功能描述

用户可以向系统上传文件，系统对大文件采用分片上传方案，支持断点续传和秒传。

### 功能要求

1. **上传初始化**
   - 接收文件名、文件大小、SHA-256、分片数量、目标目录等信息
   - SHA-256 匹配已有文件时触发秒传
   - 存在未完成任务时触发续传，返回已上传分片列表
   - 新文件创建上传任务，返回 taskId
   - 分片总数写入 Redis

2. **分片上传**
   - 接收分片文件（taskId + chunkNumber + file）
   - Redis 判重防止重复上传
   - 分片文件写入磁盘，格式：{chunkNumber}.part
   - 分片信息暂存 Redis Hash（不立即写数据库）
   - 异步更新数据库中的已上传分片计数

3. **自动合并**
   - 最后一片上传完成后自动触发合并
   - Redis 分布式锁防止并发合并
   - Redis 中读取所有分片信息批量写入数据库
   - 使用 FileChannel.transferTo 拼接分片文件
   - 合并后的文件以随机数命名
   - 自动解决同名文件冲突

4. **文件完整性校验**
   - 合并完成后异步计算完整文件的 SHA-256
   - 使用 MappedByteBuffer 分块处理大文件（每块 100MB）
   - 与前端传入的 SHA-256 比对：
     - 匹配：文件状态标记为正常，写入 SHA-256
     - 不匹配：文件名追加"（文件损坏）"，状态标记为损坏
   - 前端未传 SHA-256 时不做校验

5. **上传状态管理**
   - 上传中 / 已合并待校验 / 已完成 / 失败 / 文件损坏等状态
   - 支持续传查询已上传的分片列表

## 5.3 文件目录管理

### 功能描述

用户可以通过目录对文件进行分类管理，支持无限级层目录结构（邻接表模型）。

### 功能要求

1. **目录结构**
   - 系统自动创建：根目录 → User 目录 → 用户个人目录
   - 用户个人目录以用户名命名
   - 所有用户操作在个人目录下进行
   - 支持无限层级子目录

2. **创建目录**
   - 指定父目录和目录名称
   - 校验目录名称格式（不含非法字符 \/:*?"<>|，≤100字）
   - 同级目录自动解决重名冲突（追加编号）
   - 校验父目录存在

3. **修改目录**
   - 支持修改目录名称
   - 自动解决同级重名冲突
   - id=2 的 User 父目录不可重命名
   - 校验目录名称格式

4. **删除目录**
   - 递归软删除目录及其下所有文件和子目录
   - 不允许删除根目录（parent_id=0）
   - 不允许删除 User 父目录（id=2）
   - 不允许删除用户个人目录
   - 校验用户所有权

5. **移动目录**
   - 更新目录的 parent_id
   - 循环检测：不能移动到自身或自己的子树中（最多追溯 100 层）
   - 不允许移动根目录和 User 父目录
   - 自动解决目标目录下的重名冲突

## 5.4 文件列表查询

### 功能描述

用户可以按目录查看文件列表，系统返回目录下的子目录（全部）和文件（分页）。

### 功能要求

1. **用户个人目录内容**
   - 登录后默认展示用户个人目录下的内容
   - 返回子目录列表（按 sort 排序，全部返回）和文件列表（按上传时间倒序，分页）

2. **指定目录内容**
   - 根据目录 ID 查询其下的子目录和文件
   - 目录和文件合并计数参与分页
   - 目录排在前面（按 sort 正序），文件排在后面（按 create_time 倒序）

3. **所有目录列表**
   - 获取当前用户的所有目录（平铺列表，按父目录 + sort 排序）
   - 用于前端移动文件时展示可选目录

## 5.5 文件下载

### 功能描述

用户可以下载文件到本地。

### 功能要求

1. 根据文件 ID 查询文件信息
2. 从存储路径读取文件
3. 设置 Content-Disposition: attachment 响应头，触发浏览器下载
4. 文件名 URL 编码，支持中文文件名
5. 根据文件后缀设置对应 Content-Type
6. 检查文件状态（仅允许下载状态正常的文件）

## 5.6 文件在线预览

### 功能描述

用户无需下载即可在线查看文件内容。

### 功能要求

1. **内联预览**
   - 设置 Content-Disposition: inline，浏览器直接打开可预览的文件类型
   - 支持 50+ 种文件类型的 MIME 映射
   - 图片（jpg/png/gif/bmp/webp/svg）
   - 视频（mp4/avi/mov）
   - 音频（mp3/wav）
   - 文档（pdf/doc/docx/xls/xlsx/ppt/pptx）
   - 文本/代码（txt/java/py/js/ts/css/html/md/yaml 等）
   - 压缩包（zip/rar）

2. **kkFileView 集成**（前端）
   - 前端通过 kkFileView 服务生成预览链接
   - 预览地址格式：`http://{kkFileView地址}/onlinePreview?url={文件URL}`

## 5.7 文件删除

### 功能描述

用户可以删除自己的文件。

### 功能要求

1. 软删除：将 file_info 的 status 置为 0，物理文件保留磁盘
2. 物理文件供秒传复用，不做立即删除
3. 只允许文件上传者删除自己的文件
4. 删除操作记录日志

## 5.8 文件重命名

### 功能描述

用户可以修改文件名。

### 功能要求

1. 修改文件名（含扩展名）
2. 同步更新文件后缀字段
3. 校验文件名格式（不含非法字符，≤255字）
4. 同一目录下自动解决重名冲突（追加编号）
5. 只允许文件上传者操作

## 5.9 文件移动

### 功能描述

用户可以将文件移动到其他目录。

### 功能要求

1. 更新文件记录的 parent_id
2. 目标目录下自动解决重名冲突

## 5.10 操作日志管理

### 功能描述

记录用户对系统的操作行为，用于操作追踪和问题排查。

### 功能要求

1. 使用 AOP 切面 + 自定义 @LogRecord 注解自动记录
2. 方法执行成功后才记录日志
3. 记录字段：
   - 操作用户 ID
   - 操作类型（当前统一为 "API"）
   - 操作描述（注解中定义，如"用户登录"、"上传分片"等）
4. 支持的操作日志描述：
   - 用户登录 / 登出 / 注册 / 获取信息 / 修改信息
   - 初始化上传任务 / 上传分片
   - 获取目录列表 / 获取目录内容
   - 下载文件 / 预览文件
   - 创建目录 / 获取所有目录 / 移动目录 / 删除目录 / 重命名目录
   - 移动文件 / 删除文件 / 重命名文件

# 6 大文件上传详细需求

本章对系统的核心功能——大文件分片上传做详细的需求分析。

## 6.1 分片上传

### 需求背景

大文件（如 100MB 以上的视频、压缩包）在网络传输中存在以下问题：
- 单次上传时间长，容易超时
- 网络波动导致上传中断，需从头开始
- 失败重试代价高

### 需求描述

系统需要将大文件在客户端切割为多个小分片，逐一上传，最终在服务端合并。

### 详细要求

1. **文件切片**
   - 前端在浏览器端将文件切割为多个分片
   - 默认分片大小 5MB
   - 分片数量由文件大小和分片大小决定

2. **分片上传**
   - 支持上传单个分片到服务端
   - 每个分片携带分片编号（从 1 开始）
   - 服务端按 taskId 隔离分片存储

3. **并发控制**
   - 多个分片同时上传完成时，只能有一个触发合并操作
   - 使用 Redis 分布式锁保证合并的原子性

4. **分片判重**
   - 服务端通过 Redis 判断分片是否已上传
   - 已上传的分片直接跳过，不做重复存储

## 6.2 断点续传

### 需求背景

用户在文件上传过程中，可能因网络中断、关闭浏览器等意外情况中断上传，再次上传同一文件时需要从断点继续，而不是重新开始。

### 需求描述

系统需要支持断点续传：同一用户再次上传同一文件（通过 SHA-256 识别）时，返回已上传的分片编号，前端只上传缺失的分片。

### 详细要求

1. **任务识别**
   - 通过 SHA-256 判断是否为同一文件
   - 通过用户 ID + SHA-256 查找未完成的上传任务

2. **状态查询**
   - 在初始化接口中完成续传判断，不单独提供状态查询接口
   - 返回已上传分片编号列表（从 Redis 读取）

3. **上传跳过**
   - 前端根据已上传分片列表，只上传缺失分片
   - 服务端对已上传的分片直接判重跳过

## 6.3 秒传

### 需求背景

当系统中已有其他用户上传过相同文件（由 SHA-256 判断）时，无需重复上传物理文件。

### 需求描述

系统需要支持秒传：通过 SHA-256 判断文件是否已存在，若存在则直接创建新的文件记录引用同一物理文件。

### 详细要求

1. **文件识别**
   - 前端使用 hash-wasm 计算文件的 SHA-256
   - 服务端查询 file_info 表是否存在相同 SHA-256

2. **秒传处理**
   - 为当前用户创建新的 file_info 记录
   - 新记录引用已存在的物理文件（storage_path 相同）
   - 文件名为用户上传时的文件名（自动去重）
   - 目标目录为用户指定的目录
   - 返回 taskId = -1 标识秒传成功

3. **边界场景**
   - 不同用户上传相同文件：各自创建独立 file_info，引用同物理文件
   - 同一用户上传到不同目录：parent_id 不同，互不影响
   - 同名文件冲突：自动追加编号

## 6.4 文件合并

### 需求描述

全部分片上传完成后，服务端将分片文件合并为完整文件。

### 详细要求

1. **自动触发**
   - 不提供独立的合并接口
   - 最后一片上传完成时自动触发合并

2. **合并方式**
   - 按分片编号升序拼接
   - 使用 FileChannel.transferTo 循环写入
   - 合并后的文件名使用 5 位随机数

3. **容错处理**
   - 合并失败时删除不完整文件，事务回滚
   - Redis 锁超时自动释放（10 分钟 TTL）

## 6.5 文件完整性校验

### 需求描述

合并完成后，系统需要校验文件是否完整，即合并后的文件与原始文件内容一致。

### 详细要求

1. **校验方式**
   - 异步计算合并后完整文件的 SHA-256
   - 使用 MappedByteBuffer 分块映射（每块 100MB），避免大文件 OOM
   - 与前端传入的 SHA-256 比对

2. **校验时机**
   - 合并完成后，在事务提交后才开始计算
   - 使用独立线程池异步执行，不阻塞上传响应

3. **校验结果**
   - 校验通过：文件状态标记为正常（status=3），写入 SHA-256
   - 校验不通过：文件名追加"（文件损坏）"，状态标记为损坏（status=-1）
   - 前端未传 SHA-256：不做校验，保持待校验状态（status=1）

## 6.6 上传任务清理

### 需求背景

用户上传文件中途放弃时，临时分片文件和上传任务记录会占用磁盘和数据库空间，需要定期清理。

### 需求描述

系统需要定时清理超过 24 小时未完成的上传任务及其临时文件。

### 详细要求

1. **清理范围**
   - 超过 24 小时 status=0（上传中）的上传任务
   - 对应的临时分片文件（.part 文件）
   - 对应的 file_chunk 数据库记录
   - Redis 中的分片上传缓存

2. **状态更新**
   - 任务状态更新为 2（上传失败）

3. **执行方式**
   - 定时任务每日执行
   - 扫描条件：status=0 AND create_time < 当前时间 - 24 小时

# 8 非功能需求

## 8.1 安全性需求

1. **用户认证**
   - 采用 Sa-Token Token 认证
   - Token 有效期 1 小时
   - 未登录接口返回 401 错误

2. **用户隔离**
   - 用户只能查看和操作自己的文件和目录
   - 通过 createUser 字段与用户绑定
   - 删除/移动/重命名操作校验用户所有权

3. **密码加密**
   - 密码使用 BCrypt + Pepper 加密
   - Pepper（"uniplore"）服务端硬编码

4. **输入校验**
   - 用户名格式校验（字母开头，仅字母数字，1~50字）
   - 密码长度校验（2~100位）
   - 目录名/文件名校验（不含非法字符，长度限制）

5. **文件完整性**
   - 上传后异步 SHA-256 校验
   - 校验不通过标记文件损坏

## 8.2 性能需求

1. **大文件上传**
   - 支持单文件最大 100MB（服务端配置限制）
   - 分片上传减少单次传输量
   - Redis 暂存分片信息减少数据库写入压力

2. **并发支持**
   - 支持多用户同时上传
   - Redis 连接池最大 200 个连接
   - 分布式锁保证并发安全

3. **响应速度**
   - 分片上传接口快速返回（合并是异步触发）
   - 文件列表分页查询
   - 目录+文件合并分页设计

## 8.3 可维护性需求

1. **模块结构**
   - Controller / Service / Mapper 分层清晰
   - 工具类职责单一（CacheUtil / FileHashUtil / ValidateUtil）
   - AOP 切面统一记录操作日志

2. **配置管理**
   - application.yaml 集中管理配置（数据库/Redis/文件上传限制）
   - 开发/生产环境配置分离

3. **日志记录**
   - 自动记录所有业务操作日志
   - MyBatis-Plus SQL 日志输出
   - 分片上传详细日志（判重跳过、合并触发等）

## 8.4 可靠性需求

1. **文件完整性**
   - 合并失败时自动清理不完整文件
   - 异步 SHA-256 校验检测传输错误
   - 文件损坏标记独立状态

2. **上传容错**
   - 断点续传避免重复上传
   - Redis 分片判重保证幂等
   - Redis 锁超时自动释放

3. **数据一致**
   - 分片上传使用事务保护
   - 合并操作原子性（Redis 锁 + 事务）
   - 事务提交后才执行哈希校验
